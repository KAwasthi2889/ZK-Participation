## 🔴 CRITICAL — Demo Will Fail If Not Fixed

### 1. `cipher-bridge.ts` uses Node.js `Buffer` — Browsers Don't Have It

**Where:** `cipher-bridge.ts` lines 105, 134, 268
**What happens:** Browser throws `ReferenceError: Buffer is not defined`
**Impact:** Wallet connection fails immediately. Demo dies at step 1.

**Current code (broken):**
```typescript
const hexPrefix = Buffer.from(pk).toString("hex").slice(0, 20);  // line 105
const participantPk = fromBase64(
  Buffer.from(_addr.replace("mn1_", ""), "hex").toString("base64")  // line 134
);
const commitmentHex = "0x" + Buffer.from(commitment).toString("hex");  // line 268
```

**Fix — Replace with browser-safe helpers:**
```typescript
// Add these helpers at top of cipher-bridge.ts
function bytesToHex(bytes: Uint8Array): string {
  return Array.from(bytes, b => b.toString(16).padStart(2, "0")).join("");
}
function hexToBytes(hex: string): Uint8Array {
  const clean = hex.replace(/^0x/, "");
  const arr = new Uint8Array(clean.length / 2);
  for (let i = 0; i < arr.length; i++) {
    arr[i] = parseInt(clean.slice(i * 2, i * 2 + 2), 16);
  }
  return arr;
}
```

Then replace:
- `Buffer.from(pk).toString("hex")` → `bytesToHex(pk)`
- `Buffer.from(..., "hex")` → `hexToBytes(...)`
- `Buffer.from(commitment).toString("hex")` → `bytesToHex(commitment)`

### 3. `btoa`/`atob` corrupts cryptographic bytes ≥ 128

**Where:** `cipher-bridge.ts` lines 72, 76
**What happens:** Base64 encoding of random bytes produces incorrect values when bytes are > 127 (high bit set). This causes key derivation to produce wrong public keys, breaking the entire credential chain.
**Impact:** Silent failure — credentials don't match, proofs always return "invalid".

**Current code (broken):**
```typescript
function toBase64(bytes: Uint8Array): string {
  return btoa(String.fromCharCode(...bytes));  // ⚠️ corrupts bytes > 127
}
function fromBase64(str: string): Uint8Array {
  return Uint8Array.from(atob(str), (c) => c.charCodeAt(0));  // ⚠️ same issue
}
```

**Fix — Use URL-safe base64 (browser-native, no corruption):**
```typescript
function toBase64(bytes: Uint8Array): string {
  const bin = Array.from(bytes, b => String.fromCharCode(b)).join("");
  return btoa(bin);
}
function fromBase64(str: string): Uint8Array {
  const bin = atob(str);
  return Uint8Array.from(bin, c => c.charCodeAt(0) & 0xff);
}
```

Wait — that's the same code. The issue is that `String.fromCharCode` with values > 255 is wrong, but `Uint8Array` values are 0-255, and `btoa` expects Latin1 (0-255). Actually, `btoa` DOES handle bytes 128-255 correctly as long as the input string contains those byte values. The issue is that JavaScript strings are UTF-16, and `String.fromCharCode(...bytes)` creates a string where each char code is the byte value. This SHOULD work for 0-255.

Actually, let me reconsider. `btoa` takes a binary string (each char = 1 byte). `String.fromCharCode(...bytes)` with bytes 0-255 creates a string where each character has that code point. This IS correct for `btoa`. The problem might be with `atob` → `charCodeAt(0)` which returns the UTF-16 code unit (0-65535). For Latin1 range (0-255), this is fine.

So `btoa`/`atob` with `String.fromCharCode` actually works for bytes 0-255. The issue is more subtle: `atob` returns a string where characters are in the range 0-255. `charCodeAt(0)` returns 0-255. This is fine.

However, there's still a risk with Unicode conversion in some browsers. A safer approach is to use `URL-safe base64` encoding manually. But for the hackathon, the current code will likely work since all bytes are 0-255.

Let me downgrade this from CRITICAL to HIGH. It might work in most browsers but is fragile.

Actually, I just realized: the issue is that `btoa` in some contexts (like when called with Unicode strings) throws errors. But with `String.fromCharCode(...bytes)` where bytes are 0-255, it's fine. Let me check again...

The real issue: `String.fromCharCode(...bytes)` with a 32-byte array — each byte is 0-255. `btoa` accepts Latin1 strings. This IS correct. The only issue is if `bytes` contains values that look like surrogate pairs, but 0-255 never does.

**Verdict:** `btoa`/`atob` approach is actually CORRECT for bytes 0-255. Not a critical bug. Downgrading.

But there IS a real issue: some browsers (like Safari in certain modes) might have issues with `btoa`. A more robust approach would be:
```typescript
function toBase64(bytes: Uint8Array): string {
  const chars = "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/";
  let result = "";
  for (let i = 0; i < bytes.length; i += 3) {
    const b1 = bytes[i];
    const b2 = bytes[i + 1] || 0;
    const b3 = bytes[i + 2] || 0;
    const bitmap = (b1 << 16) | (b2 << 8) | b3;
    result += chars[(bitmap >> 18) & 63] + chars[(bitmap >> 12) & 63]
            + (i + 1 < bytes.length ? chars[(bitmap >> 6) & 63] : "=")
            + (i + 2 < bytes.length ? chars[bitmap & 63] : "=");
  }
  return result;
}
```

But that's overkill for the demo. The `btoa` approach will work.

## 🟡 HIGH — Will Cause Issues

### 4. `issueCredential` in bridge overwrites the state as `role: "patient"` even when called by sponsor

**Where:** `cipher-bridge.ts` line 159
**What happens:** When sponsor issues a credential, the bridge saves state with `role: "patient"`. If the sponsor then tries to issue another credential, their role is wrong. But the role field is never actually used for authorization in demo mode, so this is harmless.
**Impact:** Low for demo. Medium for production.

**Fix:** Remove the `role` field from state entirely, or set it correctly based on the calling context. For the demo, simply remove the `role` field — it's unused.

---

### 5. `issueCredential` derives sponsor key from the same secret as the patient

**Where:** `cipher-bridge.ts` line 132
**What happens:** In the bridge demo mode, the sponsor and patient share the same `secret` from localStorage. The sponsor's public key is derived from this same secret using `deriveSponsorPublicKey`. This means in demo mode, the "sponsor" and "patient" are technically the same person.
**Impact:** This is intentional for the demo — there's only one browser. In production, each user has their own secret.
**Verdict:** Not a bug. Design choice for single-browser demo.

## 🟢 LOW — Nice-to-Fix

### 6. `verifyProof` checks proof structure but doesn't verify the commitment matches stored state

**Where:** `cipher-bridge.ts` lines 319-329
**What happens:** The verifier only checks that the proof string looks valid (`startsWith("zkp_v1_")`), not that the commitment hash inside the proof matches any real credential.
**Impact:** In demo mode, any well-formed proof JSON passes. But the mock verifier in App.tsx already does this, so behavior is preserved.
**Fix for production:** Compare `parsed.commitment` against stored commitment hash.

---

### 7. No expiry check in `hasActiveCredential` contract circuit

**Where:** `ClinicalTrialCredential.compact` — the contract has `issueDate` in the data structure but no expiry logic in `hasActiveCredential`.
**Impact:** Credentials are "active" forever unless revoked. The bridge's `verifyProof` does check expiry (1 year), but the contract doesn't.
**Fix:** Add expiry check to contract or make it a configurable parameter.

### 8. `getCredentialHash` returns `default<Bytes<32>>` for missing credentials

**Where:** `ClinicalTrialCredential.compact` line 362
**What happens:** If a patient has no credential, this returns a zero-filled 32-byte array. The caller must check `credentials.member()` first.
**Impact:** Minor — caller should check existence before calling.
